# Product

## Register

product

## Users

People iterating on system prompts who want a small local-first tool instead of a hosted prompt platform. They are usually testing a concrete task, have a handful of examples, and want the app to do the optimization work without requiring them to learn GEPA internals first.

## Product Purpose

Super-Prompt helps users define a prompt task, evaluate candidate prompts on examples, and keep the best prompt revisions. Success means the user can go from task idea to a credible optimization run with minimal setup, clear feedback, and no hidden server or telemetry.

## Brand Personality

Practical, calm, and quietly smart. The product should feel like it predicts the next useful action, chooses safe defaults, and keeps advanced control nearby without making it the first thing users must understand.

## Anti-references

Do not feel like a research notebook that exposes every parameter up front. Avoid API-dashboard setup friction, wizard-heavy onboarding, marketing-page decoration, or UI that makes users debug model/provider behavior without guidance.

## Design Principles

- Make the next action obvious and usually correct.
- Prefer smart defaults over configuration, then expose details progressively.
- Explain model/provider problems in product language, not infrastructure language.
- Keep the local-first promise visible: browser storage, explicit provider calls, no remote telemetry.
- Preserve expert escape hatches without letting them dominate the first-run flow.

## Accessibility & Inclusion

Product UI should stay keyboard reachable, screen-reader legible, and usable with reduced motion. Color cannot be the only carrier of readiness, status, or error meaning.
